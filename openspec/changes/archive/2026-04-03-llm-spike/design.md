## Context

MVP v1 分析链路的核心问题：与其先抽帧再送多张图给 Vision 模型，不如直接将完整短视频送入支持视频输入的模型，让模型自行理解动作连贯性。这个方案更简单，上下文信息更完整，实现链路更短。

`001-system-architecture` 确定了三层架构和职责边界，但未锁定"关键帧抽取"作为唯一分析路径。本 spike 将直接视频输入模型作为首轮验证主路径，在进入 `002-api-contracts` 和 `003a-domain-models` 正式设计前，验证以下五个核心假设：

1. **结构化输出稳定性**：模型在接收完整视频后，能否稳定返回符合预期 schema 的 JSON
2. **三态判定可落地性**：`success / low_confidence / failed` 的判定是否可被 Prompt 明确约束
3. **时间戳可用性**：模型能否返回精度足够支撑 iOS 本地裁剪的 `clip.start_ms / end_ms`
4. **延迟可接受性**：单次 20 秒视频分析的平均响应时间是否在 MVP 可接受范围内
5. **成本可接受性**：单次分析的粗略 token 成本是否在 MVP 范围内

本 spike 仅做技术验证，所有产出代码放在 `backend/poc/llm-spike/`，不并入生产主链路。

---

## Goals / Non-Goals

**Goals:**

- 验证直接视频输入路径（≤ 20 秒视频）的结构化输出稳定性和 schema 符合率
- 验证三态判定逻辑的明确性（`success / low_confidence / failed` 各自触发条件是否可被 Prompt 明确约束）
- 验证 `clip` 时间戳的精度是否支撑 iOS 本地裁剪（精度目标：±1s 以内误差视为可接受）
- 测量单次 20 秒视频分析的端到端延迟和粗略 token 成本
- 生成 spike 结论报告，直接指导 `002-api-contracts` 字段设计和 `003a-domain-models` 结构定义
- 明确：若直接视频方案失败，后续应如何进入预处理优化路线（抽帧 / rep 分割）
- 可选：对照测试 Gemini 视频输入，评估质量与成本差异

**Non-Goals:**

- 不验证关键帧抽取策略（抽帧为后续优化路线，本轮不作为核心任务）
- 不验证 rep 分割逻辑
- 不实现正式 API 端点（无 HTTP 路由、无 handler）
- 不创建正式数据库模型或 migration
- 不实现 Go 服务层或正式业务逻辑
- 不实现 iOS 客户端代码
- 不验证配额系统、订阅系统
- 不做性能压测或并发测试
- 不外推到超过 20 秒的视频场景

---

## Decisions

### D1. 验证代码语言与位置

**决策**：使用 Go 编写验证脚本，放在 `backend/poc/llm-spike/` 下。

**理由**：
- Go 是后端正式实现语言，验证结果可直接参考用于正式实现
- `poc/` 目录与 `internal/`、`cmd/` 明确隔离，不会被误并入生产
- 相比 Python 脚本，Go 的类型系统便于验证 JSON schema 解析的准确性

**替代方案**：Python 脚本（更快迭代，但语言不一致，结论迁移有额外成本）→ 拒绝。

---

### D2. 主验证路径：直接视频输入

**决策**：本轮 spike 主路径为将完整视频（≤ 20 秒）直接发送给视觉模型，不进行预处理抽帧。主验证模型为 **qwen3.5-plus**。

**理由**：
- 直接视频输入保留了动作的连贯性信息，模型可自行理解动作序列，无需依赖抽帧覆盖率
- 实现链路最短，适合首轮验证
- qwen3.5-plus 支持视频输入且成本合理

**抽帧路线的位置**：抽帧与 rep 分割作为后续优化路线保留，仅在直接视频方案失败时进入。本轮不验证、不实现。

---

### D3. 输入视频限制

**决策**：Spike 样本默认使用时长 ≤ 20 秒的视频。若原始样本超过 20 秒，须先裁剪至 20 秒以内再参与验证。Spike 结论仅对 ≤ 20 秒视频有效，不外推。

**理由**：
- MVP v1 要求用户录制 3-8 次重复动作，20 秒足以覆盖典型使用场景
- 限制视频时长使 token 成本和延迟具有可预测性
- 更长视频的可行性留给后续优化迭代确认

---

### D4. 输入样本方案

**决策**：使用 2-3 段真实训练视频（≤ 20 秒），覆盖以下场景：

| 样本 | 动作 | 质量特征 | 验证目标 |
|------|------|----------|----------|
| S1 | 深蹲 | 正常角度、光线良好、有明显错误（膝盖内扣） | success 路径 |
| S2 | 高位下拉 | 正常角度、有明显错误（发力顺序） | success 路径 |
| S3 | 任意 | 角度不佳 / 遮挡 / 光线差，关键关节不可见 | low_confidence 路径 |

样本视频放在 `backend/poc/llm-spike/testdata/`，不提交至 Git（`.gitignore` 排除）。

**理由**：真实视频比合成视频更能暴露模型的实际判断边界。

---

### D5. Prompt 验证范围

**决策**：验证以下三类 Prompt 场景，合并为单一完整 Prompt 发送：

**P1 - 结构化输出 Prompt**：要求模型返回符合预期 schema 的 JSON，包含 `status`、`overall_summary`、`memory_cue`、`feedbacks[]`、`low_confidence_reason`。验证 schema 符合率（目标：≥ 90% 的调用在不使用 function calling 的情况下可直接 JSON 解析）。

**P2 - 三态判定 Prompt**：在 Prompt 中明确定义 `low_confidence` 的触发条件（至少包含：主要关节不可见、关键动作阶段未出现在视频内、角度无法判断左右侧），要求以明确规则描述，不得使用模糊表述。

**P3 - 时间戳 Prompt**：要求模型在 `clip.start_ms / end_ms` 中给出基于视频时间轴的具体时间戳，精确到秒级或亚秒级。

---

### D6. 结构化输出解析方案

**决策**：使用 **Prompt 约束 + JSON 正则提取** 方式，不强绑定特定服务商的 Structured Outputs API。

**理由**：
- 保持服务商可切换性（qwen3.5-plus / Gemini / 其他）
- 在 spike 阶段先验证纯 Prompt 方式的可行性
- 若 schema 符合率低于 80%，再考虑引入服务商特定的结构化输出能力，并在报告中记录对切换成本的影响

---

### D7. 延迟与成本验证

**决策**：Spike 须记录每次调用的端到端延迟（从发送请求到收到完整响应）和粗略 token 用量，计算单次分析的估算成本。

- **延迟通过标准**：平均延迟 ≤ 30 秒（同步请求场景下用户可接受的上限）
- **成本通过标准**：单次分析成本不超过 ¥0.20（按实际 token 定价估算）

---

### D8. 成功判定标准

Spike 整体视为**通过**，当且仅当以下条件全部满足：

| 验证项 | 通过标准 |
|--------|----------|
| Schema 符合率 | ≥ 90% 的调用（纯 Prompt 方式）可直接 JSON 解析，所有必填字段存在 |
| 三态判定 | `low_confidence` 在 S3 样本中 ≥ 2/3 被正确触发；`success` 在 S1/S2 中 ≥ 2/3 被正确触发 |
| 时间戳精度 | ≥ 2/3 的 `clip` 时间戳与人工标注误差 ≤ 1s |
| 延迟 | 主模型平均端到端延迟 ≤ 30 秒 |
| 成本 | 单次分析估算成本 ≤ ¥0.20 |

---

### D9. Spike 失败时的调整方案

**直接视频方案整体失败时**：进入预处理优化路线，在后续 change（`009-analysis-service` 阶段）引入关键帧抽取或 rep 分割预处理，再送图序列给 Vision 模型。

**具体失败项的调整建议**：

| 失败项 | 对 002-api-contracts 的影响 | 建议调整 |
|--------|----------------------------|----------|
| Schema 符合率 < 80% | `feedbacks[]` 字段结构可能需简化 | 启用服务商结构化输出 API；或减少字段数量 |
| 时间戳精度不达标 | `clip` 字段是否保留存疑 | MVP v1 降级为不返回 `clip`，iOS 不做片段裁剪 |
| 三态判定不稳定 | `status` 字段语义需要更严格约束 | 在后端增加规则引擎兜底判断（非纯 LLM 裁量） |
| 延迟超标 | 同步 HTTP 接口超时设置需重新评估 | 考虑引入异步分析任务（影响 002-api-contracts 接口模式） |
| 成本超标 | 影响 MVP 定价模型可行性 | 切换更低成本模型；或引入抽帧预处理降低 token 用量 |

**进入预处理优化路线时的后续步骤**：
1. 在 spike 报告中明确记录直接视频方案的失败原因
2. 在 `009-analysis-service` change 中引入关键帧抽取模块（均匀采样为初始策略）
3. 评估是否需要额外的 spike 验证抽帧方案

---

## Risks / Trade-offs

- **真实视频样本获取**：spike 依赖真实训练视频，若样本不具代表性，结论可信度有限 → 缓解：至少覆盖"正常错误"和"画质不佳"两类场景
- **模型输出不稳定性**：相同 Prompt 不同次调用可能返回不同结果 → 缓解：每个样本至少调用 3 次，记录结果分布
- **API 费用**：3 样本 × 3 次 ≈ 9 次调用，预计 < $5 → 可接受范围
- **时间戳验证的主观性**：人工标注的"正确"时间戳依赖判断者认知 → 缓解：标注时记录判断依据，确保可复现
- **20 秒限制的代表性**：实际用户视频可能超出 20 秒 → 本轮仅验证 20 秒以内，超长视频场景留给后续迭代
