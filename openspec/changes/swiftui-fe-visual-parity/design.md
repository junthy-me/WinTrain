## Context

当前 iOS 客户端已连通分析、历史、配额和订阅基础设施，但页面仍以原生表单结构为主，无法表达 `fe/` 中已稳定下来的视觉稿。此次改动覆盖多个页面和导航壳层，属于跨视图的 UI 架构调整，但不改变领域模型、网络契约或本地存储语义。

## Goals / Non-Goals

**Goals:**
- 在 SwiftUI 中提供与 FE 视觉稿一致的深色主题、卡片层次、头部样式和底部导航。
- 保持现有 MVP 功能链路可用，包括配额刷新、视频选择/录制、分析提交、结果展示、历史浏览、订阅购买与恢复。
- 抽出可复用的主题 token 和基础组件，避免每个页面重复拼装样式。

**Non-Goals:**
- 不修改 backend API contract、配额规则、StoreKit 产品定义或历史存储格式。
- 不引入新的业务页面或扩展 FE 尚未提供的功能。
- 不追求像素级复刻 Web 运行时特性，例如浏览器 hover 态或 Web 图片生成链路。

## Decisions

### 1. 用自定义 SwiftUI 应用壳替换默认 `TabView` 外观

保留 SwiftUI 导航能力，但不依赖系统默认 tab bar 的外观，因为默认样式无法满足 FE 的固定底栏、配色和排版要求。根视图将改为自定义 shell，根据当前选中的主 tab 承载对应页面，并在底部叠加自定义导航栏。

备选方案：
- 继续使用 `TabView` 并做外观微调。放弃，因为系统 tab bar 无法达到 FE 的视觉结构一致性。
- 完全自建路由系统。放弃，因为当前页面规模不大，仍可利用 `NavigationStack` 管理详情流转。

### 2. 建立轻量设计系统层

新增统一的颜色、圆角、间距、阴影和常用容器组件，例如页面容器、卡片、主按钮、状态徽标和列表单元，以便多个页面共享同一视觉语言。

备选方案：
- 在每个页面内联样式。放弃，因为后续维护成本高，且难以保证一致性。

### 3. 保持 ViewModel 和服务层边界不变，仅重组展示层

现有 `AnalysisFlowViewModel`、`HistoryStore`、`QuotaStore` 和订阅服务已经承担业务状态管理，本次只补足展示层需要的派生状态和导航状态，不把视觉逻辑下沉到服务层。

备选方案：
- 重新设计整个状态管理。放弃，因为会扩大变更范围并增加回归风险。

### 4. 用本地可控占位素材和现有业务数据驱动 FE 风格页面

FE 中部分页面依赖 mock 图片和演示数据。SwiftUI 版将优先复用现有业务实体；对仅影响装饰效果的演示图或预览背景，使用稳定的远程图片或本地占位视图承接，而不是改变分析结果结构。

## Risks / Trade-offs

- [风险] 自定义导航壳替换 `TabView` 后可能影响页面返回路径和环境对象注入 → 缓解：保留 `NavigationStack` 作为详情流承载，只替换外层 chrome。
- [风险] 视觉重构覆盖面大，容易引入布局回归或安全区问题 → 缓解：统一在壳层处理背景、底部内边距和滚动容器。
- [风险] FE 存在 demo 态数据，与真实 iOS 业务结果不完全一一对应 → 缓解：在不改变业务语义的前提下，定义稳定的状态映射规则。

## Migration Plan

1. 新增设计 token、基础组件和自定义主壳。
2. 逐页替换现有 `List`/`Form` 页面，接回原有 ViewModel 和 Store。
3. 运行 SwiftUI 编译验证，确认任务清单全部完成后更新 OpenSpec 任务状态。

## Open Questions

- 无阻塞性开放问题；如真机联调时发现系统相机/相册入口与视觉稿存在平台约束差异，再单独追加 change。
