# WinTrain 系统总览

## 1. 组件

WinTrain MVP v1 由三个独立组件构成：

1. iOS 客户端
2. Go 后端服务
3. 第三方 Vision LLM 服务

iOS 客户端只与 Go 后端通过 HTTPS 通信，不直接调用 LLM 服务。

## 2. 核心目标

- 让用户在一次同步请求内完成“上传视频 -> 分析 -> 展示结果”
- 保持 MVP 架构简单，不引入对象存储、异步队列、云端历史同步
- 为后续切换对象存储、异步任务、更多模型供应商预留扩展点

## 3. 系统结构

```text
+-------------------+        HTTPS         +------------------------+
| iOS Client        | ------------------>  | Go Backend API         |
| - SwiftUI UI      |                      | - Upload validation    |
| - Video capture   |                      | - Quota decision       |
| - Local history   | <------------------  | - Analysis orchestration|
| - Keychain ID     |     JSON response    | - Subscription state   |
+-------------------+                      +-----------+------------+
                                                       |
                                                       | HTTPS
                                                       v
                                             +----------------------+
                                             | Vision LLM Provider  |
                                             | - Image understanding|
                                             | - JSON output        |
                                             +----------------------+
```

## 4. 关键架构决策

### 4.1 视频上传

- MVP v1 使用客户端直传后端
- 请求格式为 `multipart/form-data`
- 后端设置 `200 MB` 硬限制
- 合约层预留 `video_source` 字段，为未来切换对象存储保留兼容性

### 4.2 分析路径

- MVP v1 主路径：后端抽关键帧后调用 Vision LLM
- 关键帧策略：均匀采样 `8-12` 帧，默认 `10` 帧
- `Gemini` 直接视频模型保留为后续补充对照，不阻塞主流程

### 4.3 部署

- Go 后端单体部署
- 使用 Docker 容器化
- 运行在单台 VPS
- 对外暴露 `/health`

### 4.4 视频存储

- 不持久化原始视频
- 仅在请求生命周期内保留临时文件
- 请求结束或服务重启后清理残留临时文件

## 5. 同步请求约束

MVP v1 保持同步请求-响应模型：

- 客户端上传视频后保持连接等待
- 后端在同一请求内完成分析并返回结构化结果
- 不引入任务队列、轮询、回调通知

同步模式的超时预算如下：

- iOS 客户端请求超时：`90s`
- 后端整体分析超时：`75s`
- 单次 LLM 提供商超时：`45s`
- 后端最多重试 `1` 次瞬时网络失败；仍失败则返回 `status=failed`

## 6. MVP 范围内与范围外

### 范围内

- 2 个动作：深蹲、高位下拉
- 结构化分析结果
- 本地历史记录
- 服务端配额裁决
- 订阅购买与恢复购买

### 范围外

- 对象存储
- 云端历史同步
- 实时分析
- 实时纠错
- 多设备历史同步
- Web 正式客户端
