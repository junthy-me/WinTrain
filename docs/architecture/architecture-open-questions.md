## 第一类：系统架构问题

### Q1. iOS 客户端的视频上传路径是什么？

**阻塞性**: ⛔ 高 — 阻塞 iOS 上传实现和后端分析入口

**候选方案**:
- **方案 A（简单）**: 客户端将视频直接 POST 到后端 API，后端转发给 LLM
- **方案 B（扩展性好）**: 客户端先上传到对象存储（如 S3/OSS），后端获取 URL 后分析

**影响范围**: iOS 网络层、后端分析 API 入口、视频存储策略

---

✅ **已决策**: MVP v1 采用客户端直传后端（方案 A）；Contract 层预留未来切换对象存储的能力。

**决策评估**:

合理性：✅ 正确。方案 A 基础设施最简单，无需引入 S3/OSS、CDN 等额外依赖，完全符合 MVP 原则。

**Contract 预留能力的具体含义**（需在 API Contract 文档中落实）：
- 分析接口定义为 `POST /v1/analysis`，请求体支持 `multipart/form-data` 上传
- 响应结构中加入 `video_source` 字段（枚举值 `direct_upload` / `storage_url`），今后切换时只需改服务端实现，客户端合约不变
- 后端内部把"接收视频"和"分析视频"解耦为两个步骤，方便日后将第一步替换为对象存储回调

**需关注的风险**:
- 视频文件可能较大（30–100 MB），直传时 iOS 网络层需要处理上传进度、超时重试、弱网中断恢复；Contract 中应规划超时参数（建议客户端设置 300 秒超时）
- 后端必须对请求体大小设上限（建议 200 MB），并在 Contract 中声明，防止异常视频撑爆服务

### Q3. 视频是否在后端持久化存储？如果是，存在哪里？

**阻塞性**: 🟡 中 — 影响隐私声明、数据架构

**候选方案**:
- **不存储**: 后端分析完即丢弃，隐私风险最低
- **临时存储**: 保留 7 天，用于排查问题，后定期清理
- **长期存储**: 保留供用户复看（但 MVP 历史记录似乎只有文字摘要）

**约束**:

**影响范围**: 后端存储设计、iOS 本地缓存策略、隐私声明文案、服务端成本

✅ **已决策**（001-system-architecture）: 后端分析完成后不持久化视频，视频仅在分析过程中存在于内存或临时目录，处理完成后立即清理。服务崩溃重启后 SHALL 清理残留临时文件。

## 第二类：动作分析核心问题

### Q4. LLM 调用方式是什么？

**阻塞性**: ⛔ 高 — 阻塞分析服务实现

**候选方案**:
- **方案 A**: 视频转为关键帧图片序列 → 发送多图给 Vision LLM（如 GPT-4o、Claude 3.5 Sonnet）
- **方案 B**: 视频直接传给支持视频输入的 LLM（如 Google Gemini 1.5）
- **方案 C**: 先用 Apple Vision 提取骨骼关键点 → 再将数据描述发给文本 LLM

**约束**:
- CLAUDE.md 说明"MVP v1 动作分析采用 LLM 优先方案"
- CLAUDE.md 提到后续可能演进为"Apple Vision + 规则引擎 + LLM 融合方案"
- 分析必须输出结构化结果（不是模糊自然语言结论）

**影响范围**: 后端分析管道、API 成本、分析延迟、结果质量

---

✅ **已决策**: MVP v1 主路径采用"后端抽关键帧 + Vision LLM"（方案 A）；直接视频模型（方案 B）仅作为对照实验，不作为首版主链路。

**决策评估**:

合理性：✅ 正确。方案 A 实现最简单，成本可控，且 Vision LLM（GPT-4o / Claude 3.5 Sonnet）对多图理解能力已验证成熟。

**关键帧抽取策略需明确**（在 LLM 集成方案验证阶段确定）：
- **抽帧数量**：建议 8-12 帧（覆盖完整动作周期）。过少会丢失关键信息，过多会增加成本和延迟
- **抽帧策略**：均匀采样 vs 动态检测（如检测动作幅度变化最大的帧）。MVP 建议先用均匀采样，简单可靠
- **图片分辨率**：建议压缩到 1280×720 或 1920×1080，平衡清晰度与 token 成本
- **成本估算**：以 GPT-4o 为例，10 张 1280×720 图片约消耗 ~2000 tokens（图片）+ ~1000 tokens（输出），单次分析成本约 $0.03–0.05

**需关注的风险**:
- 关键帧可能错过关键错误瞬间（如深蹲最低点膝盖内扣）。建议在 POC 阶段用真实视频验证抽帧策略的有效性
- Vision LLM 对动作连贯性的理解可能不如视频模型。需在 Prompt 中明确要求"根据帧序列推断动作轨迹"

**方案 B（直接视频）作为对照实验的价值**：
- 可在 POC 阶段同时测试 Gemini 1.5 Pro（原生支持视频），对比分析质量和成本
- 如果方案 B 显著优于方案 A，可在后续版本切换

### Q5. 动作分析结果的结构化输出格式是什么？

**阻塞性**: ⛔ 高 — 影响 LLM Prompt 设计、Contract 定义、iOS 展示

**前端已展示的字段（来自 fe/src/types.ts）**:
```
Feedback {
  title: string        // 问题名称（如"背部未先发力"）
  description: string  // 问题描述
  howToFix: string    // 改进方法
  cue: string         // 提示口令
}
```

**PRD 要求的分析结果结构（第 7.3 节）**:
```
- 整体结论（一句话）
- 最关键问题（1 个，必选）
- 次要问题（1-2 个，可选）
- 记忆口令（1 句）
- 最具代表性错误片段（用于复看）
```

---

✅ **已决策**: 采用"顶层结果 + 反馈项数组"双层结构；整体结论与最关键问题分为独立字段；加入 severity；错误片段返回时间戳范围；引入 `low_confidence` 中间态且不计次。

**决策评估**:

合理性：✅ 整体设计合理，双层结构清晰，各字段职责明确。

**建议的结构化输出格式**（供 Contract 文档参考）：

```json
{
  "status": "success | low_confidence | failed",
  "exercise_id": "squat | lat-pulldown",
  "overall_summary": "整体基本稳定，但发力顺序还有明显问题",
  "memory_cue": "先沉肩，后下拉；肘往下走，腋下夹紧。",
  "feedbacks": [
    {
      "rank": 1,
      "title": "背部未先发力",
      "description": "你一开始更像是在用手把杆拉下来...",
      "how_to_fix": "先想肩往下沉，再想肘往下压。",
      "cue": "先沉肩，后下拉。",
      "severity": "major | minor | info",
      "clip": {
        "start_ms": 2300,
        "end_ms": 4100
      }
    }
  ],
  "low_confidence_reason": "画面角度不足，无法可靠判断膝关节方向"
}
```

**各字段说明与补充**：

- `status` 三态：
  - `success`：正常分析完成，计次
  - `low_confidence`：画面质量不足，返回保守反馈 + 引导重拍，**不计次**（与 PRD 第 10.1 节一致）
  - `failed`：服务异常、超时等技术失败，不计次
- `severity` 取值建议：`major`（主要问题，必须优先修正）/ `minor`（次要问题）/ `info`（提示性信息）。`rank=1` 的 feedback 应始终为 `major`
- `clip` 字段：MVP v1 返回时间戳范围，iOS 端用于在本地视频上定位片段。**注意**：时间戳基于原始上传视频，后端需在分析时记录关键帧对应的时间位置
- `low_confidence_reason`：仅在 `status=low_confidence` 时存在，用于向用户展示具体原因

**需关注的风险**：
- `low_confidence` 的判断标准需要在 LLM Prompt 中明确定义（如"画面遮挡超过 30%"、"关键关节不可见"），否则 LLM 可能随意触发该状态
- `clip` 时间戳依赖后端在抽帧时记录帧时间，需要在关键帧抽取模块中同步实现
- `severity` 字段当前前端未使用，但 iOS 实现时应预留展示逻辑（如只展示 `major` 问题）

### Q6. 分析超时阈值与失败策略是什么？

**阻塞性**: 🟡 中 — 影响用户体验设计和权益扣减逻辑

**问题**:
- 用户可接受的最长等待时间是多少？
- LLM 调用超时后，是否重试？重试几次？
- 超时失败是否计次？
- 前端 AnalyzingScreen 当前模拟 6 秒（100/5% × 0.3s），实际 LLM 分析可能需要 15-60 秒

**影响范围**: 后端分析服务、iOS 等待 UI、配额扣减逻辑

✅ **已决策**:

- MVP v1 采用同步请求-响应模式
- iOS 单次请求超时设为 `90s`
- 后端整体分析预算 `75s`
- 单次 provider 调用预算 `45s`
- 后端仅对瞬时网络错误重试 `1` 次
- 超时、provider 错误、抽帧失败统一返回 `status=failed`
- `low_confidence` 与 `failed` 均不计次

## 第三类：权益与订阅问题

### Q7. 设备标识方案是什么？

**阻塞性**: ⛔ 高 — 阻塞权益系统实现

**约束**:
- MVP v1 无账户系统（CLAUDE.md 明确）
- 订阅必须"以设备维度使用"
- 必须支持"恢复购买"（CLAUDE.md）

**问题**:
- iOS 使用 `UIDevice.identifierForVendor`（重装后会变）还是 Keychain 存储的自生成 UUID？
- `identifierForVendor` 在用户卸载后重新安装会重置，如何处理恢复购买场景？
- 后端使用什么作为用户唯一键？设备 UUID vs Apple Transaction ID？

**影响范围**: iOS 设备标识模块、后端配额系统、恢复购买流程

---

✅ **已决策**: Keychain UUID 作为 `install_id`；`originalTransactionId` 作为订阅恢复锚点。

**决策评估**:

合理性：✅ 正确，这是 iOS 无账户场景下的标准做法。

**Keychain UUID 的具体行为说明**：
- iOS Keychain 数据在 App 卸载后**默认会被删除**（与 Android 不同）
- 但如果 Keychain item 设置了 `kSecAttrAccessibleAfterFirstUnlock` 且用户开启了 **iCloud Keychain 同步**，数据会在设备间同步，卸载重装后可恢复
- **建议**：将 UUID 存入 Keychain 时，设置 `kSecAttrSynchronizable = true`，允许 iCloud Keychain 同步。这样用户换设备或重装后，免费次数记录可以恢复

**⚠️ 需要注意的边界情况**：
- 用户关闭 iCloud Keychain 同步时，卸载重装会生成新 UUID，免费次数重置。这是可接受的 MVP 行为，但需要在隐私说明中告知
- 用户换新设备时，如果 iCloud Keychain 同步开启，`install_id` 会迁移到新设备，免费次数不会重置（符合预期）

**两个标识符的职责划分**：
- `install_id`（Keychain UUID）：用于配额计数（免费次数、每日限制）
- `originalTransactionId`（Apple StoreKit）：用于订阅状态恢复。恢复购买时，客户端通过 StoreKit 获取 `originalTransactionId`，发送给后端，后端将该设备标记为 Pro

**后端数据模型影响**：
- 设备表需要同时存储 `install_id` 和 `original_transaction_id`（可为空）
- 恢复购买时，后端通过 `original_transaction_id` 查找订阅记录，并关联到当前 `install_id`

### Q8. 配额逻辑在哪里判断？

**阻塞性**: ⛔ 高 — 影响系统安全性和产品信任度

**候选方案**:
- **仅服务端判断**: 客户端无法绕过，安全性高
- **客户端+服务端双判断**: 用户体验更流畅，但实现复杂

**约束**:
- CLAUDE.md："配额与权益判断必须服务端可感知"
- 意味着服务端是最终裁判，但客户端可以缓存状态提升体验

---

✅ **已决策**: 最终只由服务端裁决；客户端仅缓存短时快照用于展示。

**决策评估**:

合理性：✅ 正确，服务端裁决是唯一安全的方案，防止客户端绕过配额。

**客户端缓存快照的设计建议**：

缓存内容：
```json
{
  "free_count_used": 2,
  "free_count_total": 3,
  "is_pro": false,
  "daily_used_today": 1,
  "snapshot_at": "2026-03-24T10:00:00Z"
}
```

缓存有效期建议：**5 分钟**。理由：
- 用户在健身房使用，不会频繁切换设备
- 5 分钟内的展示误差（如显示"还剩 1 次"但实际已用完）不会造成严重体验问题
- 每次分析完成后，服务端响应中应返回最新配额状态，客户端立即刷新缓存

**配额状态的获取时机**：
- App 启动时：调用 `GET /v1/quota` 获取最新状态，刷新缓存
- 每次分析完成后：服务端在分析响应中附带最新配额状态（避免额外请求）
- 缓存过期后：下次用户进入首页时静默刷新

**服务端裁决的执行位置**：
- 在分析接口 `POST /v1/analysis` 的入口处，服务端先检查配额，不足则直接返回 `402 Payment Required`，不启动分析流程
- 客户端收到 402 时，展示订阅引导页（Paywall）

**需关注的风险**：
- 网络异常时客户端无法获取最新配额，应降级展示缓存状态，并在 UI 上提示"配额信息可能未更新"
- 并发请求场景（用户快速连续提交）：服务端需要用数据库事务或乐观锁保证配额扣减的原子性

## 第四类：数据存储问题

### Q10. 历史记录存储在哪里？

**阻塞性**: 🟡 中 — 影响 iOS 本地 vs 服务端架构选择

**约束**:
- PRD 说"保存本次记录，供下次复看"（第 5.2 节第 6 步）
- CLAUDE.md："MVP v1 不包含云同步"
- 历史记录页按订阅状态展示"完整训练历史"是 Pro 权益

**问题**:
- 免费用户：历史记录存本地还是服务端？
- Pro 用户：也是本地存储还是服务端？（MVP v1 无云同步）
- 如果都存本地，后端历史查询 API 是否需要实现？

**待确认**:
- ①MVP v1 历史记录是否完全在 iOS 本地存储（Core Data/SQLite）？
- ②后端是否记录分析日志（用于运营分析），与用户历史记录分开考虑？

✅ **已决策**（001-system-architecture）:
- ① MVP v1 历史记录完全存储在 iOS 本地（Core Data），后端不提供历史查询 API
- ② 后端记录操作日志（install_id、动作类型、分析状态、时间戳），与用户可见历史记录分离，互不依赖
- 历史记录仅保存：结构化分析结果、必要展示字段、本地派生片段引用；不保存完整原始视频
- 仅 status=success 的记录写入历史；low_confidence 与 failed 不写入历史

### Q11. 后端数据库选型是什么？

**阻塞性**: 🟡 中 — 影响 Go 项目初始化

- PostgreSQL（关系型，成熟稳定）
- SQLite（超简单，适合极小规模 MVP）
- Redis（仅用于缓存配额状态）

✅ **已决策**: PostgreSQL 作为后端主数据库；MVP v1 不单独引入 Redis。

- PRD 要求"单次录制 3-8 次重复"
- 3-8 次深蹲/高位下拉的视频约 30-90 秒
- 原始 H.264 压缩后约 30-100MB

**问题**:
- 上传前是否在 iOS 端重新压缩？
- 压缩目标大小/分辨率？
- 是否有文件大小上限？

✅ **已决策**:

- iOS 上传前进行一次导出压缩
- 目标规格：H.264，长边不超过 `1280`，优先控制到 `80 MB` 以内
- 服务端硬限制：`200 MB`
- 超过限制直接拒绝上传并提示重录
