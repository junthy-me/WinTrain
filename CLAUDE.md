# WinTrain - Claude Code 项目指令

## 1. 项目使命

WinTrain 是一款面向力量训练动作分析的 iOS 健身应用。
当前 MVP 仅支持两个动作：

- 高位下拉
- 杠铃深蹲

当前 MVP 的目标不是构建通用健身平台。
当前 MVP 的目标是让用户完成一次完整闭环：

1. 选择支持的训练动作
2. 上传训练视频
3. 获取结构化动作分析反馈
4. 在应用内查看结果
5. 保存训练记录
6. 在需要时通过订阅解锁付费使用

## 2. 事实源优先级

当信息发生冲突时，按以下优先级处理：

P0 产品事实源

4. docs/market_analysis.md （市场调研结果）

6. openspec/changes/**/proposal.md
7. openspec/changes/**/design.md

P2 参考信息

不得以实现便利性覆盖 PRD 或当前 UI 行为定义。

## 3. 产品边界

当前 MVP 约束如下：

- iOS 客户端使用 Swift 实现
- 后端为独立的 Go 远程服务
- MVP v1 不包含登录
- MVP v1 不包含云同步
- MVP v1 包含订阅支付
- MVP v1 动作分析采用 LLM 优先方案
- 后续版本可能演进为 Apple Vision + 规则引擎 + LLM 融合方案
- 在当前 change 未明确要求前，不要提前实现 v2 架构复杂度

## 4. 架构边界

Swift 客户端负责：
- UI 与用户交互
- 本地页面状态管理
- 视频采集与上传
- 分析结果展示
- 购买流程接入
- 网络请求与失败重试体验
- 后续设备端 Vision 关键点提取

Go 后端负责：
- 动作分析 API
- 分析任务编排
- LLM 调用与输出解析
- 权益与配额判断
- 订阅校验支持
- 数据持久化
- 历史记录查询
- 可观测性与服务端日志

除非 design artifact 明确批准，否则不要把后端职责挪到客户端中。

## 5. 变更控制规则

不得静默修改产品语义。

若出现以下任何一种情况，必须先更新 OpenSpec Artifact，再继续实现：

- 用户可见流程变更
- UI 语义变更
- 动作分析结果字段语义变更
- 订阅 / 权益逻辑变更
- Session 生命周期变更
- API Contract 变更
- Swift / Go 模块边界变更
- 新基础设施依赖引入
- 定价 / 配额 / 免费次数规则变更

## 6. 实现风格

优先：
- 简单、明确、可读的代码
- 稳定的 Contract
- 清晰的状态流转
- 边界明确的小任务
- 隔离性好的变更
- 渐进式交付

避免：
- 预设未来的过度抽象
- 不必要的框架引入
- 隐式魔法逻辑
- 在功能开发时进行大范围重构
- 将领域逻辑与传输层或存储层混杂
- 借机修改无关文件

## 7. 任务执行规则

编码前：
1. 阅读相关事实源文件
2. 复述任务目标
3. 明确约束与非目标
4. 确认可修改文件范围

编码中：
1. 保持在任务边界内
2. 保留既有设计意图
3. 尽早暴露不确定点
4. 不擅自发挥产品行为

编码后：
1. 总结本次修改
2. 列出风险与后续事项
3. 如有验证命令则执行验证
4. 报告实现与设计不一致之处

## 8. Review 清单

在 Review 代码时，必须评估：

- 是否符合 PRD
- 是否符合当前 UI（fe/** 为当前 UI 真相）
- 是否符合动作知识文档
- 是否符合 OpenSpec 设计
- Contract 是否稳定
- 错误处理是否完整
- 测试是否充分
- 发布与失败场景是否合理
- 引入的复杂度是否必要

## 9. MVP 领域提醒

动作分析输出必须满足：
- 结构化
- 可执行
- 易于在 App UI 中展示
- 能兼容未来混合分析方案

若期望的是结构化结果，不要只返回模糊自然语言结论。

订阅设计默认假设：
- MVP 以设备维度使用
- 采用匿名安装 / 设备标识
- v1 无账户系统
- 必须支持恢复购买
- 配额与权益判断必须服务端可感知

## 10. 文件 / 目录约定

建议含义如下：

- docs/architecture/**：系统与模块设计
- docs/contracts/**：请求 / 响应 / 状态 Contract
- openspec/specs/**：已接受的规范事实源
- openspec/changes/**：进行中的变更工件
- fe/**：当前 UI 真相与交互参考
- ios/** 或 app/**：Swift 客户端实现
- backend/** 或 cmd/, internal/**：Go 后端实现

## 11. 默认响应风格

对于较大的任务：
- 先总结理解
- 再说明下一步精确动作
- 然后再实现或 review

对于实现：
- 明确写出假设
- 不隐藏权衡
- 输出简洁且可执行

## 12. 当前 MVP 的非目标

除非明确要求，不要引入：
- 完整账户系统
- 多设备同步
- 通用动作库
- 社交 / 社区功能
- 复杂推荐系统
- 过度工程化的微服务架构
- 管理后台 UI
