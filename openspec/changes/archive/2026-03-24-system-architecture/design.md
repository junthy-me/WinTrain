## Context

WinTrain 是一款 iOS 健身动作分析应用，MVP v1 支持深蹲与高位下拉两个动作。产品定义（PRD）和前端原型已完成，但 iOS 客户端与 Go 后端均尚未启动工程实现。

当前缺少以下系统级决策的成文记录：
- 三个系统组件（iOS、后端、LLM 服务）之间如何交互
- 各模块的职责边界
- 数据从录制到展示的完整流向

本 design 确立上述决策，为后续 API Contract、LLM Spike、项目初始化等工作提供基础。

**已决策的 P0 约束**（来自 architecture-open-questions.md）：

| 问题 | 决策 |
|------|------|
| 视频上传路径 | 客户端直传后端，Contract 预留切换对象存储能力 |
| LLM 调用方式 | 后端抽关键帧 + Vision LLM（Spike 验证后定型） |
| 设备标识 | Keychain UUID（install_id）+ originalTransactionId |
| 配额裁决 | 服务端最终裁决，客户端缓存 5 分钟快照 |

## Goals / Non-Goals

**Goals:**
- 明确 iOS 客户端、Go 后端、LLM 服务三者的组件关系与交互边界
- 确定各模块的职责范围，为后续实现设立清晰边界
- 确定视频上传路径、视频存储策略、历史记录存储位置等系统级决策
- 输出三份架构文档，作为整个工程阶段的参考事实源

## Decisions

### D1. 系统组件关系：三层结构

```
┌─────────────────────────────────────────┐
│           iOS 客户端 (Swift)             │
│  视频采集 / UI / 本地存储 / 订阅支付      │
└───────────────┬─────────────────────────┘
                │ HTTPS (直传视频 / API 请求)
┌───────────────▼─────────────────────────┐
│           Go 后端服务                    │
│  分析编排 / 关键帧抽取 / 配额裁决         │
│  订阅校验 / 日志记录                      │
└───────────────┬─────────────────────────┘
                │ HTTPS API 调用
┌───────────────▼─────────────────────────┐
│           LLM Vision 服务                │
│  (GPT-4o / Claude 3.5 Sonnet 等)         │
│  接收关键帧图片序列，返回结构化分析结果   │
└─────────────────────────────────────────┘
```

**理由**：三层结构职责清晰，iOS 不直接调用 LLM（避免 API Key 暴露和成本失控），LLM 调用由后端统一管理。

---

### D2. 视频上传路径：客户端直传后端

MVP v1 iOS 客户端通过 `multipart/form-data` 将视频直接 POST 到 Go 后端，后端接收后触发分析流程。

**理由**：无需引入 S3/OSS 等对象存储，基础设施最简单，符合 MVP 原则。

**Contract 预留扩展能力**：后端内部将"接收视频"与"分析视频"解耦为两步。请求体中预留 `video_source` 字段（枚举：`direct_upload` / `storage_url`），未来切换时后端实现变更，客户端合约不变。

**备选方案**：客户端先传对象存储（方案 B）。排除原因：MVP 阶段需要额外的 IAM 配置、预签名 URL 生成服务、CDN 等，复杂度不值得。

---

### D3. 后端部署形态：单体 + Docker + VPS

Go 后端以单体服务形式部署，使用 Docker 容器化，运行在单台 VPS（如阿里云 ECS / Hetzner）。

**理由**：MVP 流量规模小，单体服务运维成本最低，Docker 保证环境一致性，VPS 成本可控。Serverless 方案（Lambda 等）冷启动延迟对视频分析场景不友好。

---

### D4. 视频存储策略：分析完即丢弃，端侧优先

后端接收视频后，仅用于当次分析，**分析完成后不持久化视频**。视频数据不落盘到任何长期存储。

**理由**：
- PRD 第 10.4 节："默认端侧优先；如需云端处理，应明确说明视频用途与存储规则"
- 用户视频含有个人隐私，不存储是最低风险的选择
- MVP v1 历史记录只保留文字摘要和结果数据，不需要视频回放

**边界**：后端可在内存或临时目录中处理视频（关键帧抽取），处理完成后立即清理。

**保留选项**：如果后续版本需要视频片段回放，该决策可修订，届时需要更新隐私声明。

---

### D5. 历史记录存储：iOS 本地，后端不提供历史查询 API

MVP v1 的历史记录存储在 iOS 设备本地（Core Data / SQLite），后端不提供历史记录查询 API。

**理由**：
- CLAUDE.md 明确："MVP v1 不包含云同步"
- 本地存储实现最简单，无需额外 API 设计
- PRD 中订阅权益"可查看历史分析记录"在 MVP v1 理解为：免费用户仅看最近 N 条，Pro 用户无限制（均在本地实现）

**后端职责**：后端仍记录每次分析的操作日志（用于运营分析和排障），但这与用户可见的历史记录分离，不对客户端暴露。

**待确认（P1，不阻塞本 change）**：Q10 历史记录存储的权益细节（免费用户查看 N 条 vs Pro 无限制）留给 005-entitlement-and-subscription-design 明确。

---

### D6. 模块职责边界

**iOS 客户端**（Swift / SwiftUI）负责：
- 视频录制（AVFoundation）与上传（URLSession）
- 全部 UI 与页面状态管理
- 本地历史记录存储（Core Data）
- 订阅购买（StoreKit 2）
- 配额状态缓存（5 分钟快照）
- 设备标识（Keychain UUID）

**Go 后端**负责：
- 接收视频，触发分析任务
- 关键帧抽取（FFmpeg 或 Go 原生库）
- LLM 调用与结构化输出解析
- 配额裁决（服务端最终权威）
- 订阅校验（Apple S2S 通知或 receipt 验证）
- 操作日志记录
- 权益状态维护

**LLM Vision 服务**（第三方）负责：
- 接收关键帧图片序列
- 返回结构化动作分析结果（JSON）

**明确不允许越界的行为**：
- iOS 客户端不得调用 LLM API（避免 Key 泄露）
- iOS 客户端不得在本地执行配额裁决（仅展示缓存快照）
- 后端不得在响应中返回视频二进制数据（端侧优先原则）

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 视频直传后端，大文件（>100MB）上传超时 | iOS 设置 300s 超时；后端限制请求体 200MB；可在上传前客户端压缩至目标大小（Q14 决策后执行）|
| 后端单点故障（单体 VPS） | MVP 阶段可接受；准备 Docker Compose 快速重部署脚本；健康检查 + 告警 |
| 视频不存储导致无法排查分析质量问题 | 后端保留分析请求的元数据日志（视频 MD5、时长、动作类型），不保留视频本身；如需排查，联系用户重新提交 |
| 历史记录仅本地存储，换设备后丢失 | MVP v1 可接受；在应用内说明；后续版本可引入云同步 |
| LLM 服务商单点依赖 | 后端 LLM 调用层抽象为接口，方便切换服务商，不在本 change 实现 |
