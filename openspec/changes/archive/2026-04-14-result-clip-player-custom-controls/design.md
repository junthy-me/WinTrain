## Context

结果页的问题片段回放只服务于短视频片段复盘，业务需求集中在基础播放控制和少量 UI 文案定制。系统 `VideoPlayer` 的优势是实现快，但它的倍速菜单文案、倍速项和菜单结构不可控，已经无法满足当前产品要求。

## Goals / Non-Goals

**Goals**
- 为问题片段播放页提供可控的倍速菜单与基础播放控制。
- 保持底层播放能力基于系统 `AVPlayer`，避免重新实现解码或视频渲染能力。
- 将自定义范围限制在结果页片段回放场景，不扩展为全局播放器框架。

**Non-Goals**
- 不复刻系统播放器全部能力，例如 AirPlay、PiP、字幕、多音轨与复杂手势体系。
- 不改变问题片段来源、导出逻辑或分析结果结构。
- 不新增第三方多媒体依赖。

## Decisions

### 1. 使用 `AVPlayer` 作为播放内核，覆盖自定义控制层

继续依赖系统 `AVPlayer` 保证稳定性与性能，界面层改由 SwiftUI 自行渲染。视频画面使用系统播放器层承载，交互按钮、进度条和倍速菜单由业务侧控制。

备选方案：
- 保留 `VideoPlayer` 并尝试局部改文案。放弃，因为系统控制层不可定制。
- 引入第三方播放器库。放弃，因为当前需求范围小，外部依赖成本不成比例。

### 2. 控制层只实现片段复盘需要的最小集合

首批能力限制为：
- 完成关闭
- 播放 / 暂停
- 前进 / 后退 10 秒
- 当前进度拖动
- 当前时间 / 总时长展示
- 倍速菜单

这能覆盖当前产品要求，同时把维护面控制在较低水平。

### 3. 倍速菜单采用业务自定义列表

菜单标题使用中文“倍速”，选项顺序固定为：
`0.25x`、`0.5x`、`1x`、`1.25x`、`1.5x`

选择后直接修改 `AVPlayer.rate`，并在恢复播放时保持当前倍速。

## Risks / Trade-offs

- [风险] 自定义控制层需要自己维护播放状态同步与时间监听 → 缓解：控制范围限制在最小功能集，集中处理 observer 生命周期。
- [风险] SwiftUI 与播放器层桥接可能出现 dismiss 后资源未释放 → 缓解：在页面消失时统一暂停并移除时间监听。
- [风险] UI 复杂度上升 → 缓解：仅在结果页片段回放场景使用，不抽象为通用播放器框架。

## Migration Plan

1. 先落 OpenSpec 工件，明确播放器改造范围。
2. 在 `ResultView.swift` 中用轻量自定义播放器替换现有 `VideoPlayer`。
3. 保持现有问题片段首帧封面逻辑不变，只替换播放页内控制层。
4. 编译验证并更新任务状态。
